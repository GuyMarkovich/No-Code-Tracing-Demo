# Kubernetes .NET RabbitMQ Distributed Tracing Demo

This demo shows distributed tracing across two standard .NET 8 services connected by RabbitMQ, using only Kubernetes-native primitives:

- No OpenTelemetry Operator.
- No RabbitMQ Operator.
- No OpenTelemetry NuGet packages or manual tracing initialization in the C# apps.
- .NET auto-instrumentation is injected through an init container and CLR environment variables.
- Each application pod has a local OpenTelemetry Collector sidecar that forwards traces to a centralized Tempo instance.
- Grafana is pre-provisioned with a Tempo datasource.

## Files

- `k8s/namespace.yaml`: isolated `tracing-demo` namespace.
- `k8s/collector-configmap.yaml`: sidecar OpenTelemetry Collector config.
- `k8s/rabbitmq.yaml`: single-node RabbitMQ `StatefulSet` and `Service`.
- `k8s/observability.yaml`: Tempo + Grafana deployment, config, and services.
- `k8s/service-a.yaml`: producer Web API deployment and service.
- `k8s/service-b.yaml`: consumer Worker Service deployment.

## Before Deploying

Replace these image placeholders with your .NET 8 application images:

- `ghcr.io/example/tracing-service-a:latest`
- `ghcr.io/example/tracing-service-b:latest`

The manifests assume the apps can be configured with these environment variables:

- `RABBITMQ_HOST=rabbitmq`
- `RABBITMQ_PORT=5672`
- `RABBITMQ_USERNAME=demo`
- `RABBITMQ_PASSWORD=demo`
- `RABBITMQ_QUEUE=orders`

If your apps use different configuration keys, update only the Kubernetes environment variables. No C# code changes are needed.

## Deploy

```powershell
kubectl apply -f k8s/namespace.yaml
kubectl apply -f k8s/collector-configmap.yaml
kubectl apply -f k8s/rabbitmq.yaml
kubectl apply -f k8s/observability.yaml
kubectl apply -f k8s/service-a.yaml
kubectl apply -f k8s/service-b.yaml
```

Wait for everything to become ready:

```powershell
kubectl -n tracing-demo rollout status statefulset/rabbitmq
kubectl -n tracing-demo rollout status deployment/observability
kubectl -n tracing-demo rollout status deployment/service-a
kubectl -n tracing-demo rollout status deployment/service-b
```

## Generate A Trace

Port-forward Service A:

```powershell
kubectl -n tracing-demo port-forward svc/service-a 8080:80
```

In another terminal, call the producer endpoint. Adjust the path to match your Web API route:

```powershell
curl http://localhost:8080/publish
```

Port-forward Grafana:

```powershell
kubectl -n tracing-demo port-forward svc/grafana 3000:3000
```

Open Grafana at `http://localhost:3000`. Log in with:

- Username: `admin`
- Password: `admin`

Go to **Explore**, select the **Tempo** datasource, and search recent traces. Open the trace generated by the Service A request.

## What To Validate In Grafana

In the trace waterfall, you should see a single trace that crosses the asynchronous RabbitMQ boundary:

- An inbound HTTP server span in `service-a`.
- A RabbitMQ publish span from `service-a` with messaging attributes such as `messaging.system=rabbitmq`, `messaging.operation=publish`, or related RabbitMQ destination/routing metadata.
- A RabbitMQ receive/process span in `service-b` with messaging attributes such as `messaging.system=rabbitmq` and a consumer operation.
- The `service-b` span should be part of the same trace ID as the original `service-a` HTTP request.

That shared trace ID is the proof point: the .NET auto-instrumentation injected W3C trace context into RabbitMQ message headers on publish, and extracted it on consume, without modifying the application code.

If you see separate traces for producer and consumer, confirm the apps use a RabbitMQ client version supported by the .NET auto-instrumentation image and that the message headers are preserved by the publish/consume flow.
